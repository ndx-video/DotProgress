# General - Flatten OKF frontmatter fields

## Summary

Flattened optional OKF metadata into top-level frontmatter fields so entries no longer use a `research` parent object. Updated documentation and skill guidance accordingly, and changed the example entry type value to `progressive_agent_log`.

## Changes

- Updated `.progress/README.md` to define flat top-level OKF fields and require `type` when optional OKF fields are present
- Updated `README.md` to describe flat OKF metadata usage
- Updated `.claude/skills/dotprogress/SKILL.md` to instruct agents to emit flat OKF fields
- Updated `.progress/general/G000003.example-research-metadata-entry.md` by flattening metadata and setting `type: progressive_agent_log`

## Decisions

- Use flat frontmatter keys for easier parsing across repositories and tooling
- Keep all OKF metadata optional except `type` when optional OKF metadata is included

## Follow-ups

- Consider documenting a controlled vocabulary for `type`, `okf_class`, and `okf_capability` values
